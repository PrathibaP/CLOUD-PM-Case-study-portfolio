01
Problem Statement

The Permission Creep Crisis

Cloud environments grant permissions far faster than they are ever revoked. Over time, IAM roles, users, and service accounts accumulate hundreds of permissions they never use — a condition called permission creep. This silently expands the blast radius of any breach: a compromised credential with excessive permissions can delete databases, exfiltrate terabytes, or laterally move to other services with no additional steps.

Security teams know this problem exists. But the native AWS console surfaces raw policy JSON — not actionable risk signals. Engineers drown in documents. Auditors produce static spreadsheets that go stale immediately. Fixes get deprioritized because there's no workflow that connects finding a problem to fixing it fast.

⚠️
Root cause: No closed-loop workflow exists — from permission discovery, to risk prioritization, to one-click remediation — that fits into an engineer's existing day. The gap between knowing permissions are excessive and actually remediating them is precisely where organizations get breached.

03
Proposed Features & Wireframe Rationale

Three Screens. One Closed Loop.

The product is designed as a linear workflow: Discover → Investigate → Fix. Each screen maps to exactly one stage, eliminating context-switching and keeping the remediation cycle short. Every UI decision is justified by the primary persona's cognitive load.

1

Risk Dashboard

Entry point giving security engineers an immediate risk snapshot. 4 KPI tiles surface critical signal at a glance. The Top Risk Entities table ranks identities by risk score with one-click drill-down — no search required for daily triage. The AWS service heatmap shows where excess permissions cluster. A 30-day trend chart provides time-series context absent in point-in-time audits.

→ View Screen 1
2

Permission Explorer

The analytical core. Engineers filter, search, and drill into any identity to see what permissions exist, which haven't been used in 90+ days, and what specific dangerous actions (e.g., s3:DeleteBucket, iam:PassRole) are granted. The "Reduce to minimum" button auto-generates a least-privilege policy from CloudTrail logs — this is the product's core value proposition and differentiator.

→ View Screen 2
3

Remediation Queue

Where IAM tools most often fail: they find problems but don't help fix them. This screen introduces a structured queue with three fix types: Auto-fix (1-click with approval), Manual (human judgment required), and Review (snooze with audit trail). Policy diffs, Terraform export, Jira integration, and a live progress bar that creates accountability.

→ View Screen 3
💡
Phase 2 (Roadmap): Blast Radius Graph View
A D3.js force-directed graph visualizing identity → policy → permission inheritance chains. Shows who granted what, through which policy, enabling lateral movement risk analysis for incident response teams. Deprioritized for MVP due to build complexity (Dev Item L).

06
Bonus: Development Action Items

13 Dev Action Items for Engineering

Organized into 5 engineering tracks. Each item is scoped for sprint-level planning and maps directly to a prioritized feature.

🔵 Track 1 — Data Ingestion & Analysis
Dev Item A

CloudTrail Integration

Ingest 90-day API call logs per identity. Normalize event names to IAM permission strings. Handle multi-region, compressed (gzip) S3 log delivery. Rate-limit S3 reads for large orgs.

HIGH effort
Dev Item B

IAM Policy Parser

Flatten managed + inline + permission boundary + SCP policies into a unified permission graph per identity. Handle policy conditions, variables, and NotAction/Deny semantics correctly.

MED effort
Dev Item C

Least-Privilege Generator

Diff granted permissions vs. CloudTrail-observed usage → output minimal policy JSON. Validate via IAM policy simulator before surfacing. Flag edge cases (infrequent but legitimate use).

HIGH effort
🟡 Track 2 — Backend & API
Dev Item D

Risk Scoring Engine

Weighted model: action sensitivity × days-unused × identity type × blast radius. Must be customer-configurable (e.g., stricter thresholds for prod vs. staging accounts).

MED effort
Dev Item E

Multi-Account Support

Cross-account IAM role assumption via AWS Organizations. Single-pane-of-glass across all accounts. Handle 10,000+ identities with pagination and background processing.

HIGH effort
Dev Item F

Scheduled Scan Engine

Configurable scan frequency (daily/weekly). Delta detection between scans to drive trend charts. Queue-based async processing. Alert on sudden risk score spikes.

MED effort
🟢 Track 3 — Remediation & Integrations
Dev Item G

Auto-fix Apply Engine

Apply generated policies via AWS SDK with IAM policy simulator pre-flight validation. Require approver confirmation before write. Full rollback support. Audit log every mutation.

HIGH effort
Dev Item H

Jira Integration

Create tickets per manual remediation item: pre-populated summary, assignee (policy creator), priority mapping, due-date based on risk. Sync fix status back from Jira.

LOW effort
Dev Item I

Terraform / CDK Export

Generate HCL/CDK diffs per policy change. Output as downloadable .tf file or a PR-ready patch. Support both Terraform AWS provider and CDK v2 IAM constructs.

MED effort
🔴 Track 4 — Security & Compliance
Dev Item J

Approval Workflow

2-person approval required for auto-fix on production accounts. RBAC per account tier. Full immutable audit log of all policy mutations made via the platform. SOC2 evidence export.

MED effort
Dev Item K

Snooze Audit Trail

All snooze actions logged with: actor, reason, expiry timestamp. Auto-resurface at expiry with escalation. Export log to SIEM (Splunk/Datadog). Never silently drop a finding.

LOW effort
🎨 Track 5 — UX / Front-end
Dev Item L

Graph View (Phase 2)

D3.js force-directed graph: identity → policy → permission inheritance. Essential for blast-radius analysis in incident response. Must virtualize for graphs with 500+ nodes.

HIGH effort
Dev Item M

Real-time Scan Feedback

WebSocket-based progress streaming during active scans. Display partial results as they arrive. Never block the UI. Show per-account scan ETA for large org scans.

LOW effort
